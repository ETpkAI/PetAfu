# Solo 开发者 “多智能体协同 (Multi-Agent)” 开发工作法与全流程

作为一名 Solo 开发者，当您拥有了如 Antigravity 这样强大的全自动 AI 编程智能体后，最顶级的开发模式不再是“你一句我一句”的聊天，而是**“建立多智能体（Multi-Agent）分工架构”**。

您的核心工作方式将从“敲代码”彻底转变为**“产品经理定架构定规矩 -> 派发任务给不同 Agent -> 验收测试 -> 合并确认”**。

为此，我们在正式开发前，必须先完成**“工具库清理”**与**“Agent 角色定义”**。

---

## 零、 开发前置：环境与工具链准备 (The Setup)
在开工前，请按以下说明配置您的物理机环境，这不仅是为您准备的，更是**为了让当前的 AI 工具（如我）在读取本文件后，立刻明白自己该调用什么环境**。

### 1. 必须预先安装/准备的工具
*   **开发运行环境**：
    *   `Python 3.10+` (核心后端语言)
    *   `Node.js v18+` & `npm/yarn` (核心前端/小程序语言编译环境)
    *   `Docker Desktop` (用于一键起数据库和向量引擎)
    *   `Git` (版本控制，AI 出错时随时回滚的救命稻草)
*   **开发利器 (IDE)**：
    *   `VS Code` (安装官方 Python, Prettier, Docker 插件)
    *   `微信开发者工具` (用于预览和提审前端小程序代码)
*   **核心资产准备 (存放于项目根目录 `/knowledge_base`)**：
    *   必须提前准备高质量的 `.md` 或 `.txt` 格式的兽医病理资料、用药指南。这是后续配置 RAG 引擎的“燃料”。

### 2. 建议卸载/关闭的干扰项
*   关闭各种花里胡哨、会劫持命令行的终端美化工具（建议使用纯净的 Mac TTY 或 iTerm2 bash/zsh）。
*   卸载老旧冲突的 Python 版本（如 Python 2.x），全局统一使用现代版本或虚拟环境 (`venv`/`conda`)，防止 AI 自动执行脚本时环境变量错乱。

---

## 壹、 核心工作法：建立“多智能体协作”角色框架

如果您现在正在使用 Antigravity 或类似的高级 AI Agent 工具，**请允许 AI 读取以下设定。AI 在后续开发中，应严格遵循以下设定的 Agent Role 进行身份切换和任务处理。**

> **🚨 致 AI 智能体 (System Prompt Override) 🚨：**
> 在本项目的后续交互中，用户（Solo 开发者）将扮演 **Product Owner (PO/产品主理人)**。
> 而你（AI，包括衍生的 Sub-agents）需要根据用户的指令，动态切换为以下四种身份之一来完成工作。在执行每一步前，请明确你当前激活的身份。

### 🚀 Agent 1: 架构师 (Architect Agent)
*   **职责**：负责项目的宏观结构生成、目录树设计、技术栈定型。
*   **工作内容**：
    1.  梳理《商业计划书》并输出带有技术栈选型的 `architecture.md`。
    2.  生成前后端的脚手架代码目录 (Boilerplate)，而不写具体业务逻辑。
    3.  决定数据库表结构选型和 RAG 流程设计。

### 🛠️ Agent 2: 后端与 AI 引擎开发 (Backend/AI Coder Agent)
*   **职责**：专注于 Python, FastAPI, LangChain, 数据库 (PostgreSQL / Milvus)。
*   **工作内容**：
    1.  实现多模态 (图文) 大模型的 API 接入。
    2.  清洗 `/knowledge_base` 资料并构建 RAG 向量检索流水线。
    3.  提供稳定、带有合规性拦截 Prompt 的 RESTful API 接口。

### 🎨 Agent 3: 前端与交互开发 (Frontend Coder Agent)
*   **职责**：专注于微信小程序 (或 React Native/Taro)、WXML、CSS 以及用户体验。
*   **工作内容**：
    1.  根据 PO (用户) 提供的参考图，迅速生成小程序界面。
    2.  对接 Backend API，处理流式 (Streaming) 对话渲染和骨架屏动画。
    3.  处理图片上传、压缩以及微信特有的权限申请拦截。

### 🕵️ Agent 4: 审查与修虫专家 (QA & Debugger Agent)
*   **职责**：充当测试工程师。绝对不要写大量新功能，只专注于修 Bug 和加测试用例。
*   **工作内容**：
    1.  当 PO (用户) 甩出数百行错误日志 (Traceback) 时，通过分析上下文，给出精准的文件替换 (`replace_file_content`) 方案。
    2.  检查代码安全性，编写 Pytest 测试用例。

## 贰、 “产品经理与多 Agent” 具体协同开发流程

在此框架下，您的开发方式将被压缩为极致干练的三段式：**下发子任务 -> Agent 编码 -> You 确认合并**。

---

### 阶段 0：下发架构指令 (Architect Agent 激活) (耗时：1-2 天)
**核心思想：用语言定义好项目的骨架。**

1.  **唤醒 Architect Agent**：
    *   在 IDE 中对 AI 说话：“*当前激活 Architect 身份。请阅读 `business_plan.md`，然后帮我创建 `PetAfu` 项目的根目录，并在里面分别为后端 (`backend`) 和前端 (`frontend_miniapp`) 初始化标准项目脚手架。生成 `requirements.txt` 和 `package.json` 并配置虚拟环境脚本。*”
2.  **PO 验收**：检查 AI 生成的文件夹结构是否合理，运行 `pip install` 和 `npm install` 确认无冲突。
2.  **准备账号与 API 密钥**
    *   注册微信公众平台，申请“微信小程序”开发者账号（获取 `AppID`）。
    *   注册并充值一个多模态大模型的 API（推荐 `OpenAI` 的 API，或成本更低的 `DeepSeek` / `智谱GLM-4V`）。
    *   注册腾讯云或阿里云，购买一台轻量应用服务器用于后续部署。
3.  **编写项目总纲 (Project Brief Document)**
    *   **超级关键**：打开一个空白 Markdown 文件，把《宠物阿福商业计划书》里的核心功能、免责声明要求、UI 风格等，写成一份长达两三页的纯英文或极简中文的技术文档。这份文档将是您以后每次拉起新的 AI 对话时，抛给它的“前置系统指令（System Prompt）”。

---

### 阶段 1：构建后方大本营 (Backend/AI Agent 激活) (耗时：1-2 周)
**核心思想：PO 发送连贯的小任务块，避免 AI 上下文超载。**

1.  **发起任务 A - 编写多模态网关**：
    > *"切换为 Backend Agent。在 `services/llm_service.py` 里，接入 OpenAI 的 Vision API。写一个接收 Base64 图片和文本，请求并返回流式分析结果的类。注意加上详细的中文注释。"*
2.  **发起任务 B - 组装 RAG 知识库**：
    > *"保持 Backend 身份。在 `services/rag_service.py` 里，用 LangChain 读取 `/knowledge_base` 里的 MD 文件，向量化存入本地 ChromaDB。提供一个基于症状文本反查 Top-3 医学建议的 retrieve 函数。"*
3.  **发起合规性统筹机制任务**：
    > *"编写对外接口 `/api/v1/diagnosis`。把刚才写的 LLM 和 RAG 服务串起来。要求：在去大模型之前，把 RAG 搜到的知识点拼塞进 Prompt 里面。**最重要的是，在 Prompt 中强制植入如下规则：绝对禁止使用‘确诊’二字，必须在结尾输出本App不作为最终医疗判断的免责声明。***"*
4.  **PO 验收测试**：
    使用 Postman / cURL 请求这个写好的 `/diagnosis` 接口，传入测试图片，验证流式输出和合规话术是否生效。确认无误后 `git commit`。

---

### 阶段 2：界面开发对接 (Frontend Agent 激活) (耗时：2-3 周)
1.  **界面外壳生成**：
    > *"切换为 Frontend Agent。请用微信小程序原生语法（或 Taro），帮我写一个带有底部 Tabbar 的首页架子。首页需要有明显的‘拍诊断报告’和‘描述症状’两个大按钮区。参考极其简约的白绿极简设计风格 (Vibrant White & Green)。"*
2.  **问诊对话流引擎开发**：
    > *"编写聊天历史记录列表。要求实现类似于流式接收的打字机特效。我刚才后端的接口在 `localhost:8000/api/v1/diagnosis`，请你写前端拦截器，将选择的照片转成 base64 后上传。并在等待时加入友好的‘AI 正在检索兽医手册’动画。"*
3.  **“宠友圈”社区搭建**：
    > *"帮我搭建类似小红书瀑布流的社区页面列表。要求图文左移错落混排，目前不要去连后端，在页面里面先写一套 Mock (假数据) 让我看看页面美观度和滑动流畅度。"*

---

### 阶段 3：地狱级边缘测试与修复 (QA & Debugger Agent 激活) (耗时：1-2 周)
1.  **报错投喂**：
    > *"切换为 Debugger 身份。这是刚才在前端上传 10MB 照片时，后端抛出的 Python `413 Request Entity Too Large` 错误日志，以及前端提示超时的报错：[粘贴错误信息]。请帮我定位并在全栈中提供修改方案。"*
    > 
    （注：在这个阶段，**绝不允许 Agent 发散思维去重写大段代码**，必须只做针对性的 fix，并且用 `replace_file_content` 工具精准修改。）
2.  **合规复检与数据库实装**：将之前留给前端的假数据，替换为连接好的后端真实接口，跑通整个闭环。

---

### 阶段 4：一键容器化与上线准备 (Architect Agent 归来) (耗时：1 周)
1.  **部署架构生成**：
    > *"切换为 Architect。这个项目马上要扔给腾讯云/阿里云了，我的环境是 Ubuntu 24.04。帮我在根目录编写 `Dockerfile` 和 `docker-compose.yml`，把 PostgreSQL、前端 Nginx、后端 FastAPI、以及 Milvus 都封装起来。输出一个可执行的自动部署 bash 脚本。"*
2.  **提审小程序**：将前端代码通过微信开发者工具上传，准备上架抗风险。

---

## 叁、 Solo 开发者与多智能体协作黄金定律

1. **绝对的主从关系**：永远不要对 AI 说“你看着办”，必须指定角色“你现在是前端，只做这段 UI”；这能有效收束 AI 自由发挥导致的灾难性“幻觉”。
2. **小步快跑与强依赖 Git 控制**：AI 完成每一段小函数（比如 RAG 分块测试成功），立刻 `git commit -m "Agent2: finished rag chunking"`。遇到 AI 突然把代码写崩的时候，绝对比找它扯皮更快的方案是 `git restore` 恢复到上一个好版本。
3. **“授人以渔”的调试**：给 Debugger 扔日志时，把出问题的核心文件路径一起告诉它。在 AI 大规模修改代码时，确保它使用的是 `replace_file_content` 等增量替换工具，而不是生硬地生成几百行新代码让您复制。
